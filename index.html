2d arpg前端

cocos2d-xARPG前端技能系统设计

概述

以下设计思路可以用于类传奇，也可以用于部分arpg

数据表的设计

考虑到此种类型游戏技能比较复杂，一个技能可能分数种阶段（项目是6种）带数种特效，产生数种后继状态。技能和buf需要分开，两者的特效也要分开。初步设计为5张表处理技能。skillData处理技能数据skilleffect处理特效数据，一条skillData数据可以与薯条skilleffect信息关联。BufferData处理buffer，Buffereffect处理buf特效。一条skilldata可以与数条buf关联。

  • skilldata
  • skilleffect
  • bufferdata
  • buffereffect

资源的处理

原则如下：

  • 图像资源用10位数字分类区别类型2，id3，动作2，方向1和帧数，如有必要i可以增加位数
  • 单方向技能每个方向分开合图，8方向分为8个文件，需要时再加载
  • 多方向技能合并成一个文件，利于合批降低drawcall
  • 资源管理器设置技能资源子管理器，子管理器中分数个管道，分别处理不同资源。主角技能资源优先缓存，低等级常用公用资源缓存，其他角色技能资源换场景随即清理。

技能消息的处理

两种选择，接受消息后处理和发送消息处理。如果接受消息后处理，一旦网络延迟卡顿现象会比较严重；一般技能动画在发出消息时候同时播放，接受回包时候处理技能cd。血量和状态处理由额外的消息去做。

第一章-cocos2d-x4.0简介

Cocos2d-x4.0和之前版本的区别

iOS/macOS 支持 metal - metal是iOS和Mac独有的底层图形库，可以更好挖掘苹果设备的潜力
使用 CMake，删除各平台的工程文件 - 减少乱七八糟的文件，使用cmake工具需要时再生成项目文件
升级 GLFW 到 3.3 - openGL的开源库升级
升级 minizip 到 1.2 - 压缩库升级
删除废弃函数 - 让代码更简洁，这种简洁一般为了后继开发，估计后继会有大版本的升级。
删除 h5 引擎和 Javascript 绑定 - js h5 支持 creator
删除 tiff - 不必要的格式支持
删除 SimpleAudioEngine - 删除声音库
删除 experimental 名字空间 - 整洁代码
修复 macOS 15 系统字体绘制问题 -bug
适配 iOS13 -适配问题
UIWebView 使用 WKWebView 实现
VideoPlayer 使用 AVPlayerController 实现
修复 lua 工程在 64 位设备的崩溃问题

总的来说，Cocos2d-x4.0 和 cocos2d-x3.0比较，修复了bug，代码没有本质的区别，并不是2.x to 3.x的升级，而是对3.x的精简。删除了项目文件使用cmake生成项目文件，删除了冗余的废弃代码，3.x可以部分使用2.x的api，但是4.0x已经行不通了。3.x的资料对4.0x很多仍然适用。

移动设备常用的资源处理方式

移动设备内存有限，为了节省内存，一般都会对贴图进行压缩。
一张4MB的贴图，在移动设备中压缩之后只有1/8，也就是0.5MB。

假如有一张1024px * 1024px的4通道(RGBA8888，每个通道都为8bit)贴图。
一般来说，一张纹理所占用的内存的计算方式为长乘以框再撑每个像素的比特数，计算公式日下：

内存 = 长 * 宽 * 像素比特数

那么这张图所长的内存为：1024 * 1024 * 4 * 8 bit = 1024 * 1024 * 4 Byte = 4MB

即一个像素占 32bit == 4Byte。

一 Android设备

Android设备中一般使用ETC1压缩，一种有损的图像压缩方式。
ETC1是OpenGL2.0支持的标准，压缩之后每个像素占4bit，压缩之后的格式为KTX或者PKM，前者支持存储多纹理，后者只支持单纹理。
上面的图片压缩之后的大小为：1024 * 1024 * 4 bit = 0.5MB
ETC1的缺点是不支持Alpha通道，不支持有透明度的图片压缩。
ETC2解决了Alpha通道，但是它是OpenGL3.0标准，过去考虑到2.0设备的市场占有率，一般使用ETC1。现在市场上大部分手机支持3.0所以ETC2推荐使用

二 iOS设备

iOS设备中采用的图像格式一般是PVR，也是一种有损的图像压缩方式。

PVR压缩分为两种，PVRTC2和PVRTC4。
PVRTC4：Compressed format, 4 bits per pixel, ok image quality
PVRTC2：Compressed format, 2 bits per pixel, poor image quality

常用的PVRTC4，压缩之后的大小为：1024 * 1024 * 4 bit = 0.5MB
除了压缩内存，PVR可以直接被显卡读取，载入速度更快；缺点是PVR需要PowerVR芯片支持，目前iOS设备都能完美支持，Android支持尚少；此外，PVRTC4只支持方形贴图，非方形会被处理成方形，且长宽必须为2的幂。
现在pvrtc二代iphone8以下会出现问题，现阶段推荐使用pvrtc一代
